Java & Architecture Spring Boutique

Typage, POO & Architecture en couches

Objectifs du cours :

  • Revoir le typage en Java (primitifs, références, génériques).
  • Comprendre la POO en Java (classes, objets, encapsulation, héritage, polymorphisme).
  • Analyser l’architecture du projet Spring “boutique” et le rôle de chaque package : config, controllers, dtos, entites, mappers, repositories, services.

Projet étudié : application Spring Boot de gestion de boutique (clients, produits, commandes, paiements…).

Rappel : le typage en Java

Typage statique et fort

  • Typage statique : le type des variables est connu à la compilation.
    • Ex : int age = 25;, String nom = "Dupont";
    • Les erreurs de type sont détectées à la compilation (et non à l’exécution).
  • Typage fort : Java ne convertit pas les types “magiquement”.
    • On doit faire des casts explicites ou utiliser des méthodes de conversion.

Avantages

  • Sécurité : moins d’erreurs de type à l’exécution.
  • Lisibilité : le type exprime l’intention du développeur.
  • Outils : meilleure autocomplétion et refactorings dans l’IDE.

Types primitifs et types références

Types primitifs

  • Numériques entiers : byte, short, int, long
  • Numériques flottants : float, double
  • Booléen : boolean
  • Caractère : char
  • Stockés directement en mémoire, taille fixe, très performants.

Types références (objets)

  • Classes : String, Client, Commande, etc.
  • Interfaces et types abstraits.
  • Collections : List<Produit>, Map<K,V>, etc.
  • Les variables contiennent une référence vers l’objet dans le tas (heap).

Les génériques en Java

  • But : avoir un code réutilisable et typé (collections, services, mappers).
  • Exemple simple :
    • List<String> noms = new ArrayList<>();
    • La liste ne contiendra que des String.
  • Dans un projet Spring, on les retrouve partout :
    • Interfaces Spring Data : JpaRepository<Client, Long>
    • DTOs paramétrés (si besoin) et méthodes de services génériques.

POO en Java : concepts de base

  • Classe : “plan de construction” décrivant un type d’objet.
    • Ex : Client, Produit, Commande dans ton package entites.
  • Objet : instance concrète d’une classe.
    • Ex : un client précis : new Client("Durand", "Alice").
  • Méthodes : actions que l’objet peut réaliser.
  • Attributs : données que l’objet transporte (nom, prix, date, etc.).

POO en Java : principes clés

Encapsulation

  • Masquer les détails internes de la classe et exposer une interface propre.
  • Pratique :
    • Attributs private
    • Accesseurs getters / setters

Héritage

  • Une classe peut hériter d’une autre : class ProduitEnPromo extends Produit
  • Permet de factoriser le code commun.

Polymorphisme

  • La même méthode peut se comporter différemment selon l’objet concret.
  • Ex : une méthode afficherInfos() sur Client et sur Produit sera appelée de la même façon mais n’aura pas le même comportement.

POO & Spring dans le projet “boutique”

  • Spring repose fortement sur la POO :
    • Classes annotées (@Controller, @Service, @Entity…).
    • Interfaces pour les repos (AppartenirRepository, ClientRepository, etc.).
    • Injection de dépendances (DI) pour composer les objets entre eux.
  • Ton projet est organisé en couches (architecture en couches).

Architecture globale du projet “boutique”

Packages principaux :

  • config : configuration de l’application (par ex. SecurityConfig).
  • controllers : exposition des API REST (couche présentation / web).
  • dtos : objets échangés avec le client (front-end, Postman, etc.).
  • entites : modèles métier persistés en base de données.
  • mappers : conversion entre entites et dtos.
  • repositories : accès aux données (DAO) via Spring Data JPA.
  • services : logique métier, orchestration des opérations.
  • BoutiqueApplication : classe de démarrage Spring Boot.

Package config

Exemple : SecurityConfig.java

  • Configure la sécurité Spring (Spring Security) :
    • Gestion de l’authentification / autorisation (qui peut faire quoi).
    • Configuration des filtres, des règles pour les URLs (ex : /api/admin/** protégé).
    • Stratégies de mot de passe, CORS, etc. (selon ta config réelle).
  • But : centraliser les réglages techniques pour ne pas les mélanger avec la logique métier.

Package controllers

Exemples : ClientController, ProduitController, CommandeController, etc.

  • Couche la plus proche du client HTTP (navigateur, Postman, front Angular/React...).
  • Responsabilités :
    • Exposer des endpoints REST (ex: GET /clients, POST /produits).
    • Recevoir les données en DTOs de requête (ex : ClientRequestDTO).
    • Appeler les services pour effectuer les opérations métier.
    • Retourner des DTOs de réponse (ex : ClientResponseDTO, CommandeResponseDTO).
  • Ne doivent pas contenir de logique métier complexe : ils déléguent aux services.

Package dtos (exemple avec ClientDTO)

Exemples : ClientDTO, ClientRequestDTO, ClientResponseDTO, ProduitRequestDTO, etc.

  • DTO = Data Transfer Object.
  • Objets utilisés pour transporter des données entre :
    • Le client (front) et le serveur (controller).
    • Différentes couches internes, si nécessaire.

Dans ton projet, ClientDTO est un record Java :

public record ClientDTO(
    String nom,
    String prenom,
    String ville,
    String codePostal,
    String adresse,
    String email,
    String telephone
) {}
  • Ce record représente un client tel qu’il est exposé par l’API (couche DTO), pas forcément exactement comme dans l’entité.
  • Le typage fort garantit :
    • Que chaque champ est bien du bon type (String pour email, telephone, etc.).
    • Que les méthodes qui manipulent un ClientDTO ne peuvent pas se tromper de structure.

Package entites

Exemples : Client, Produit, Commande, Paiement, Categorie, etc.

  • Représentent les objets métier persistés en base de données.
  • Annotées avec @Entity, @Id, @ManyToOne, etc. (JPA/Hibernate).
  • Contiennent :
    • Les attributs métier (nom, prix, stock, date, etc.).
    • Les relations (ex : un Produit appartient à une Categorie).
  • Directement gérées par les repositories pour la persistance.

Package mappers

Exemples : ClientMapper, ProduitMapper, CommandeMapper, etc.

  • Responsables de la conversion entre :
    • Entités <=> DTOs
  • Intérêts :
    • Centraliser la logique de mapping dans un seul endroit.
    • Éviter de répéter le même code dans les controllers ou services.
    • Faciliter l’évolution du modèle (ajout/suppression d’attributs, renommages).

Package repositories

Exemples : ClientRepository, ProduitRepository, CommandeRepository, etc.

  • Interfaces Spring Data (souvent extends JpaRepository<Entite, TypeId>).
  • Représentent la couche d’accès aux données (DAO).
  • Responsabilités :
    • CRUD : save, findById, findAll, deleteById, etc.
    • Requêtes dérivées des noms de méthode (ex : findByEmail).
  • Spring génère l’implémentation automatiquement, tu codes surtout l’interface.

Package services

Exemples : ClientService, ProduitService, CommandeService, etc.

  • Contiennent la logique métier de l’application.
  • Responsabilités :
    • Valider les données, appliquer les règles métier.
    • Appeler un ou plusieurs repositories.
    • Utiliser les mappers pour passer de DTO à entité et inversement.
  • Les controllers appellent les services, les services appellent les repositories.

Schéma : flux d’une requête dans le projet

Exemple : création d’un client ou d’un produit.

[ Client HTTP (navigateur / Postman / front) ]
                    |
                    v
        +---------------------------+
        |   controllers             |
        |   (ex: ClientController) |
        +---------------------------+
                    |
        (reçoit un ClientRequestDTO)
                    |
                    v
        +---------------------------+
        |   services                |
        |   (ex: ClientService)    |
        +---------------------------+
                    |
        (utilise ClientMapper pour
         passer de DTO -> Entité)
                    |
                    v
        +---------------------------+
        |   entites                 |
        |   (ex: Client)           |
        +---------------------------+
                    |
                    v
        +---------------------------+
        |   repositories            |
        |   (ex: ClientRepository) |
        +---------------------------+
                    |
        (persistance en base de données)
                    |
                    v
        +---------------------------+
        |   services                |
        +---------------------------+
                    |
        (Entité -> ClientResponseDTO via mapper)
                    |
                    v
        +---------------------------+
        |   controllers             |
        +---------------------------+
                    |
        (renvoie le DTO au client HTTP)
                    v
[ Réponse JSON vers le front ]

POO & typage dans cette architecture

  • Chaque couche est composée de classes Java typées :
    • ClientService dépend de ClientRepository, ClientMapper et des différents DTOs.
    • Les relations entre objets (services, repos, entités) sont des dépendances typées.
  • La POO permet de :
    • Modéliser clairement le métier (entités).
    • Isoler les responsabilités (controllers vs services vs repositories).
    • Réutiliser le code (mappers, services partagés, etc.).

Conclusion

  • Java fournit un typage statique fort qui sécurise le développement.
  • La POO (classes, objets, encapsulation, héritage, polymorphisme) est au cœur de Java et de Spring.
  • Ton projet “boutique” illustre une architecture en couches claire :
    • config pour la configuration technique.
    • controllers pour les endpoints REST.
    • dtos, entites, mappers pour structurer les données.
    • services pour la logique métier.
    • repositories pour l’accès aux données.
